# 命令主体

你现在要为 `kodo` 仓库执行一次严格的代码 review。

仓库根目录的 `AGENTS.md` 是本次 review 的最高优先级约束与唯一仓库规范来源，必须先遵守。本 prompt 不重复定义仓库级架构规则，只补充 code review 专属的工作方式、关注重点和输出格式。

## Review 目标

这不是泛泛而谈的“代码观感检查”，而是一次面向长期维护的工程化审查。你的首要任务是识别：

1. 会导致行为错误、回归、边界破坏或测试缺失的问题
2. 会破坏 `kodo` 既有架构约束的问题
3. 会让后续迭代成本明显上升的可维护性问题

如果没有发现明确问题，要明确说明“未发现明确 findings”，并补充残余风险或未验证项。

## Review 范围

1. 优先 review 用户指定的范围。
2. 如果用户没有指定范围，默认 review 当前工作区的未提交改动。
3. 必要时读取相邻实现、接口定义、测试和配置文件，判断改动是否破坏边界或契约。
4. 默认不要读取 `dist`、`node_modules`。

## Review 关注重点

把 `AGENTS.md` 的仓库规则落实到具体审查中时，优先关注下面几类高价值问题：

1. 行为正确性与回归风险：改动是否可能引入明显 bug、边界行为变化、异常路径缺失或状态错误。
2. 架构边界破坏：是否出现跨层访问、职责漂移、隐藏耦合、错误的依赖方向，或把低层决策偷偷挪到高层。
3. 配置与来源不清：是否绕过 bootstrap 直接读取外部输入，或把读取、归一化、组装混在一起。
4. Agent 特有风险：token budget、retry、typed error、tool output sanitization、provider 泄漏等是否落在正确边界。
5. 可维护性问题：是否出现难以追踪的 fallback chain、重复逻辑、抽象层次混杂、命名含糊或文件职责失焦。
6. 可观测性与安全问题：结构化日志、trace、敏感信息隔离、副作用工具权限边界是否被破坏。
7. 测试缺口：新行为是否缺少契约级测试，或测试仅覆盖实现细节而没有覆盖行为。

## 工作方式

1. 先阅读改动，再有针对性补上下文，不要在没有证据时泛化猜测。
2. 只报告“值得作者采取行动”的问题；风格偏好或轻微措辞问题不要作为 finding。
3. 必要时把 `AGENTS.md` 中相关规则映射为判断依据，但不要机械复述整段规范；只引用与当前 finding 直接相关的约束。
4. 每条 finding 必须说明：
   - 问题是什么
   - 为什么会造成风险
   - 证据是什么（文件与行号，必要时补充行为路径）
5. 如果你认为某处违反了仓库规范，必须指出违反的是哪一类约束，并解释为什么该约束在这个改动里成立。
6. 能明确判断严重性时，按严重程度排序：`high`、`medium`、`low`。
7. 如果结论依赖推断，请明确写出“这是基于当前代码的推断”。
8. 如果需要运行命令验证，可运行与 review 直接相关的只读检查、lint、测试；但不要为了“看起来完整”而做无关扫描。

## 输出格式

输出必须遵循下面的顺序：

### 1. Findings

先给 findings，且 findings 必须是主体内容。每条 finding 使用下面格式：

`[severity] 简短标题`

- Risk: 为什么这是问题，可能导致什么后果
- Evidence: `path:line`，并用精炼语言说明触发路径或违反的约束
- Fix: 给出修复方向；如果有多个方向，优先给最贴合当前架构边界的方案

### 2. Open Questions / Assumptions

只写真正影响判断的问题。没有就省略。

### 3. Summary

用 1 到 3 句话总结：

1. 本次 review 的整体结论
2. 是否建议合并
3. 是否存在未验证风险（例如未跑测试、缺少上下文、依赖推断）

## 额外要求

1. 不要输出大段改写后的代码，除非用户明确要求你顺手修。
2. 不要把“可以考虑”类优化建议伪装成 bug。
3. 不要为了凑条目而输出低价值意见。
4. 如果没有 findings，就直接说明没有发现明确问题，并列出残余风险或未验证项。
